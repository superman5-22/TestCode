1 なぜテストコードが必要か？

木材基幹システムでは、「樹種」「等級」「含水率」「寸法（長×幅×厚）」など、計算ロジックが非常に複雑です。さらに、法改正による税率変更や、仕入れ価格の変動も激しい世界です。

  • ● デグレード（先祖返り）の防止： 計算式を修正した際、「ヒノキの価格は直ったが、杉の価格が狂った」という事故を瞬時に検知します。
  • ● 「手動確認」からの解放： 毎回画面を開いて100パターンの数値を入力するのは不可能です。テストコードなら数秒で終わります。

2 題材：木材価格計算ロジック

すべての言語で、以下のビジネスロジックをテストします。

【入力項目】

樹種（杉・檜）、寸法（ミリ単位）、枚数。および、データベースから取得する「最新の立米単価」。

【計算ロジック】

体積(m3) = (長/1000 * 幅/1000 * 厚/1000) * 枚数
金額 = 体積 * 立米単価

【依存関係（モック対象）】

在庫DB（InventoryRepository）。テスト時はDBに接続せず、「杉の単価は60,000円」という仮の結果を返させます。

☕ Java (JUnit5 + Mockito)

エンタープライズの王道
// 業務ロジック (Service)
public class LumberService { private final LumberRepository repo; // 依存性の注入（DI）でモックを差し込みやすくする public LumberService(LumberRepository repo) { this.repo = repo; } public long calculatePrice(String type, double l, double w, double t, int count) { long unitPrice = repo.getUnitPrice(type); // DB通信を想定 double volume = (l * w * t / 1_000_000_000.0) * count; return (long) (volume * unitPrice); } }
// テストコード (Test)
@ExtendWith(MockitoExtension.class) class LumberServiceTest { @Mock LumberRepository repo; // 偽物のDBを作成 @InjectMocks LumberService service; @Test void testPrice() { // 杉なら単価6万を返すよう設定（モック） when(repo.getUnitPrice("Cedar")).thenReturn(60000L); long price = service.calculatePrice("Cedar", 4000, 105, 105, 10); assertEquals(26460, price); // 検証 } }

🖥️ フロントエンドのテスト

木材システムの場合、ブラウザ上で「厚み105mm、幅105mm」と入力した際、リアルタイムでプレビューが出るような機能にテストが必要です。

推奨ツール： Vitest（ロジック）、Playwright（画面操作）
理由： 入力ミスは受注ミスに直結するため、重要な画面だけは自動で動かしてチェックすべきです。

4. 何をテストするべき？領域別の判断基準 🧩

🎨 フロントエンド

結論: テストすべき。特に「表示条件」「入力バリデーション」「主要導線」。
書くべきテスト: Unit(Vitest/Jest) + コンポーネントテスト + 重要導線E2E(Playwright/Cypress)。

🛠️ バックエンド

結論: 最優先で書くべき。業務ロジックは壊れると影響が大きい。
書くべきテスト: Unit中心 + API/DB連携のIntegration + 認証・決済導線のE2E。

🗄️ SQL / DB

結論: 書く価値が高い。JOINや集計、マイグレーション失敗は事故になりやすい。
書くべきテスト: クエリ検証、制約、マイグレーションの再現テスト。

☁️ インフラ / クラウド

結論: 「コード化(IaC)しているならテストする」。
書くべきテスト: Terraform validate/plan、静的チェック、監視アラートの動作確認。

5. フロントのテストは本当に不要？ 🖥️

答え: 不要ではありません。 UIは変更頻度が高く、体験品質に直結します。
ただし、全部E2Eにすると重くなるので「役割分担」が大切です。

Vitest / Jest
関数・Hooks・状態管理の速い確認
React Testing Library
ユーザー視点でUIの挙動確認
Playwright / Cypress
ブラウザで実際の導線を保証

代替手段として、StorybookのVisual Regression（見た目差分）を使う方法も有効です。UI崩れ検知に強いです。

6. バックエンドは先にテスト？後でテスト？ ⚖️

🧪 TDD（先にテスト）

  • Red: 失敗するテストを書く
  • Green: 最小実装で通す
  • Refactor: きれいに整える

メリット: 設計が引き締まる。仕様がブレにくい。

🛠️ After実装（後でテスト）

  • まず機能を作って動かす
  • 壊れたら困る部分からテスト追加
  • バグ再発防止テストを積む

メリット: 開発初速が出る。既存PJで導入しやすい。

初心者へのおすすめ: まずはAfter実装で始め、慣れたら新規機能でTDDを試すハイブリッドが現実的です。

7. 実際の書き方（主要7言語） 💻

以下は「テスト対象コード → テストコード」をセットで掲載。まずは構文の完全理解より、Given-When-Then（前提・操作・期待）の形を真似しましょう。

Java（JUnit5 + Mockito + Testcontainers）

// Production
class PriceService { int total(int price, int qty){ return price * qty; } }
// Test (JUnit5)
@Test void total_works(){
  var s = new PriceService();
  assertEquals(300, s.total(100, 3));
}
// Mockito example: external dependency mock
// Testcontainers example: PostgreSQLコンテナを起動してRepositoryを検証

Go（testing + testify + sqlmock + httptest）

// Production
func Total(price, qty int) int { return price * qty }
// Test
a := assert.New(t)
a.Equal(300, Total(100,3))
// sqlmock: DBクエリ期待値検証
// httptest: APIハンドラのHTTPレスポンス検証

Python（pytest + pytest-mock + requests-mock）

# production
def total(price, qty):
    return price * qty

# test
def test_total():
    assert total(100, 3) == 300

# pytest-mock: mocker.patch(...)
# requests-mock: 外部HTTPの固定レスポンス化

TypeScript / JavaScript（Jest + RTL + Cypress/Playwright）

// production
export const total = (price:number, qty:number)=> price * qty;
// unit test (Jest)
test('total', ()=>{ expect(total(100,3)).toBe(300); });
// React Testing Library
render(<LoginButton />); expect(screen.getByRole('button')).toBeInTheDocument();
// Playwright/Cypress: login→購入完了までのE2E

C#（xUnit / NUnit / MSTest + Moq + Testcontainers）

// production
public int Total(int price, int qty) => price * qty;
// xUnit
[Fact]
public void Total_Works(){ Assert.Equal(300, Total(100,3)); }
// Moq: 依存サービスをモック
// Testcontainers: DB付き結合テスト

Rust（cargo test + mockall + sqlx + reqwest）

// production
fn total(price:i32, qty:i32)->i32 { price * qty }
// test
#[test]
fn total_works(){ assert_eq!(300, total(100,3)); }
// mockall: trait依存のモック
// sqlx: DBテスト、reqwest: API統合テスト

C言語（GoogleTest / Unity / CppUTest）

// production
int total(int price, int qty){ return price * qty; }
// GoogleTest(C++)ラッパ例
TEST(TotalTest, Works){ EXPECT_EQ(300, total(100,3)); }
// 組み込みならUnity/CppUTestが軽量で使いやすい
