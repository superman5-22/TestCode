## 1. テストコードの基本概念とメリット

### 1-1. テストコードとは

| 用語 | 説明 |
|------|------|
| **テストコード** | アプリ本体（プロダクションコード）が正しく動くことを自動で確認するためのプログラム |
| **自動テスト** | コマンド一発で全テストを実行し、結果（成功/失敗）を判定する仕組み |
| **テストケース** | 「ある入力に対してある出力が返るはず」という1件の検証単位 |
| **アサーション** | 期待値と実際値を比較して合否を判定する命令（例: `assertEquals(期待値, 実際値)`）|

### 1-2. テストコードのメリット

| メリット | 説明 |
|----------|------|
| **デグレ（退行）防止** | 改修後に既存機能が壊れていないことをすぐ確認できる |
| **仕様の文書化** | テストコード自体が「このメソッドはこう動く」という仕様書になる |
| **設計の改善** | テストしにくいコードは設計が悪い証拠。テスト容易性を意識すると自然と疎結合になる |
| **リファクタリングの安心感** | テストが通れば動作を壊していないと確認できる |
| **CI/CDとの連携** | GitHub Actions等と組み合わせ、プッシュのたびに自動検証できる |

## 2. テストの種類と粒度

| 種類 | 対象範囲 | 速度 | 主な用途 |
|------|----------|------|----------|
| **単体テスト（Unit Test）** | クラス/メソッド単体 | 非常に速い | ロジックの細かな検証 |
| **統合テスト（Integration Test）** | 複数コンポーネントの連携 | 普通 | Controller + Service + Repositoryの連携確認 |
| **E2Eテスト（End-to-End Test）** | ブラウザ〜DB全体 | 遅い | 画面操作レベルの最終確認 |

> **推奨比率（テストピラミッド）**
> 単体テスト : 統合テスト : E2Eテスト ≈ **70 : 20 : 10**

## 3. 新規作成時 vs 既存コード改修時のアプローチ違い

### 3-1. 新規作成時（TDD推奨）

| ステップ | 作業内容 | ポイント |
|----------|----------|----------|
| **① 失敗するテストを書く（Red）** | まだ実装がないのでテストは必ず失敗する | 「何を実装すべきか」を先にテストで定義する |
| **② 最小限の実装でテストを通す（Green）** | ベタ書きでもいいのでテストを通す | きれいなコードは後回し、まず動かすことを優先 |
| **③ リファクタリング（Refactor）** | テストが通った状態でコードを整理する | テストが安全網になるので安心してリファクタできる |
| **④ ① に戻る** | 次のテストケースを追加する | この繰り返しが TDD サイクル（Red → Green → Refactor）|

**新規作成時のテスト観点チェックリスト**

| 観点 | 例 |
|------|----|
| 正常系（Happy Path） | 正しい入力で期待通りの結果が返る |
| 異常系（Error Path） | 不正な入力で適切な例外/エラーが返る |
| 境界値（Boundary） | 最大値・最小値・0・null など境界での挙動 |
| 空データ | リストが空のとき、nullのとき |

---

### 3-2. 既存コード改修時（Safety-Net テスト推奨）

| ステップ | 作業内容 | ポイント |
|----------|----------|----------|
| **① 現状の挙動をテストで記録（特性テスト）** | 改修前に「今の動き」をテストで固定する | バグがあっても一旦 現状の挙動 をテストに書く |
| **② 改修後もそのテストが通ることを確認** | テストが安全網として機能する | 意図しない変更を検知できる |
| **③ 要件に合わせてテストを更新・追加** | 新しい仕様に合わせてテストも変更する | テストが仕様変更の記録になる |
| **④ 不要になったテストは削除・修正** | 古い仕様のテストは削除してよい | テストも「生きたドキュメント」として維持する |

**改修時の特有リスクと対策**

| リスク | 対策 |
|--------|------|
| 既存機能の破壊（デグレ） | 改修前に十分なテストを揃えてから改修する |
| テスト対象の依存関係が多い | Mock/Stub で外部依存を切り離してテストする |
| レガシーコードでテストが書けない | 段階的リファクタリングで徐々にテスト可能な設計にする |

## 4. Java/Spring 代表的テストフレームワーク

| フレームワーク | 役割 | 主なアノテーション/メソッド |
|----------------|------|---------------------------|
| **JUnit 5** | テストの実行エンジン。テストクラス・メソッドを定義し、アサーションを行う | `@Test`, `@BeforeEach`, `@AfterEach`, `@ParameterizedTest`, `assertEquals()`, `assertThrows()` |
| **Mockito** | 依存オブジェクト（DB, 外部API等）を偽物（Mock）に差し替える | `@Mock`, `@InjectMocks`, `when().thenReturn()`, `verify()` |
| **Spring Boot Test** | Spring のコンテキストを使ったテスト。Controller等の統合テスト | `@SpringBootTest`, `@WebMvcTest`, `@DataJpaTest`, `MockMvc` |
| **AssertJ** | 可読性の高いアサーション記法を提供（JUnit のアサーションより表現豊か） | `assertThat(actual).isEqualTo(expected)`, `.contains()`, `.hasSize()` |
| **Hamcrest** | マッチャーベースのアサーション（MockMvcと相性が良い） | `is()`, `hasSize()`, `containsString()` |

### 4-1. JUnit 5 よく使うアノテーション一覧

| アノテーション | タイミング | 用途 |
|----------------|-----------|------|
| `@Test` | テストメソッドに付ける | そのメソッドをテストとして認識させる |
| `@BeforeEach` | 各テスト前 | テストデータの初期化など |
| `@AfterEach` | 各テスト後 | リソースの解放など |
| `@BeforeAll` | 全テスト前（1回のみ） | DB接続など重い初期化 |
| `@AfterAll` | 全テスト後（1回のみ） | DB接続解除など |
| `@DisplayName` | テストメソッドに付ける | テスト名を日本語等で分かりやすく表示 |
| `@ParameterizedTest` | テストメソッドに付ける | 複数の入力値でテストを繰り返す |
| `@Nested` | 内部クラスに付ける | テストをグループ化して階層構造で整理 |
| `@ExtendWith` | クラスに付ける | 拡張機能を追加（例: `MockitoExtension.class`）|

### 4-2. Mockito よく使うパターン

| パターン | コード例 | 説明 |
|----------|----------|------|
| モックの作成 | `@Mock OrderRepository repo;` | 偽のRepositoryを用意 |
| 戻り値の設定 | `when(repo.findAll()).thenReturn(list);` | 呼ばれたときの返り値を固定 |
| 例外のスロー設定 | `when(repo.findById(0)).thenThrow(new RuntimeException());` | 例外を発生させる |
| 呼び出し検証 | `verify(repo, times(1)).save(any());` | メソッドが指定回数呼ばれたか確認 |
| 呼ばれていないことの検証 | `verify(repo, never()).delete(any());` | 呼ばれていないことを確認 |

## 5. アノテーション早見表（Spring Boot Test）

| アノテーション | 読み込む範囲 | 主な用途 |
|----------------|-------------|----------|
| `@SpringBootTest` | アプリ全体 | 統合テスト全般 |
| `@WebMvcTest(XxxController.class)` | Web層のみ | Controller の単体テスト |
| `@DataJpaTest` | JPA関連のみ | Repository の単体テスト |
| `@MockBean` | Spring コンテキスト内のBean をMock化 | `@WebMvcTest` と組み合わせて使う |
| `@AutoConfigureMockMvc` | MockMvc の自動設定 | `@SpringBootTest` + HTTP テスト |

## 6. テストコードの命名規則（推奨）

| パターン | 形式 | 例 |
|----------|------|----|
| **Given-When-Then** | `given_[条件]_when_[操作]_then_[期待結果]` | `given_validOrder_when_save_then_returnsSavedOrder` |
| **日本語メソッド名** | `@DisplayName("正常系：注文が正しく保存される")` | 日本語でテスト名を記述（JUnit 5 推奨） |
| **should パターン** | `should_[期待動作]_when_[条件]` | `should_throwException_when_quantityIsNegative` |

## 7. テストコード品質チェックリスト

| チェック項目 | OK の状態 |
|--------------|-----------|
| テストは独立している | 他のテストの結果に依存していない |
| 1テスト1アサーション（原則）| 1メソッドで検証する内容は1つに絞る |
| テスト名が分かりやすい | 何をテストしているか名前だけで分かる |
| AAA パターンに従っている | Arrange（準備）→ Act（実行）→ Assert（検証）の3段構成 |
| 外部依存をモック化している | DB・HTTP通信をモックにして高速・安定したテストになっている |
| 境界値を網羅している | null, 空文字, 0, 最大値, 最小値を考慮している |
